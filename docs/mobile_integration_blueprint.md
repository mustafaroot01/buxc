# 📄 مخطط تكامل تطبيق الموبايل (Mobile Integration Blueprint)

هذا الملف مخصص لمطور تطبيق الموبايل لشرح الحلول التقنية للمشاكل المواجهة وضمان المزامنة الصحيحة.

---

## 1. آلية برمجية لمطابقة الـ QR (Offline Matching)
السيرفر يرسل بيانات الطلاب مشفرة بـ (SHA256) لحماية البيانات ولتسهيل المطابقة.

- **المفتاح المستلم من الـ API:** هو حقل `qr_hash`.
- **المدخل من الكاميرا:** نص خام (Raw Text) من الـ QR Code.

### 💡 الحل البرمجي في التطبيق:
عند قيام المعلم بمسح الرمز، يجب على التطبيق القيام بما يلي:
1. تحويل النص الممسوح إلى SHA256 Hash.
2. مقارنة الـ Hash الناتج مع الـ `qr_hash` الموجود في قاعدة بيانات الموبايل.

**مثال بالـ Logic (Pseudo-code):**
```dart
string scannedData = "STU-9823-XYZ"; // القيمة من الكاميرا
string hashedData = sha256.convert(utf8.encode(scannedData)).toString();

if (hashedData == student.qr_hash) {
   // تم العثور على الطالب بنجاح
}
```

---

## 2. أسماء حقول الطلاب (Student Fields)
تنبيه: لا يوجد حقل باسم `name`. قاعدة البيانات تعتمد التقسيم الثلاثي لضمان الدقة:
- `first_name`: الاسم الأول.
- `second_name`: اسم الأب.
- `last_name`: اللقب/اسم الجد.
- `full_name`: حقل مساعد للقراءة فقط (Read-only).

---

## 3. استجابة السيرفر بعد المزامنة (Sync Feedback)
عند إرسال طلب المزامنة إلى `/api/v1/attendance/sync` باستخدام `POST`:
- إذا نجحت المزامنة: ستستلم `status: success`.
- إذا وجد السيرفر أخطاء في بعض المسحات (طالب غير موجود أو كود متكرر)، سيعيد لك `status: partial` مع مصفوفة `errors` توضح `student_id` ونوع الخطأ.

---

## 4. ضمان عدم تكرار البيانات (Idempotency)
يجب توليد `request_id` فريد (UUID) لكل عملية مسح تتم في الموبايل.
- السيرفر يستخدم هذا المعرف لمنع تكرار تسجيل الحضور لنفس الطالب في حال تم إرسال الطلب مرتين بسبب ضعف الإنترنت.

---

## 5. مزامنة البيانات (Version Control)
- استخدم حقل `version` عند جلب الطلاب.
- أرسل آخر `version` استلمته في الطلب القادم تحت مسمى `since_version` لجلب التحديثات فقط (Incremental Sync).
